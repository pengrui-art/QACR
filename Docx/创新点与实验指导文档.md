# Query-Adaptive Compute Routing 研究创新点与实验指导文档

## 1. 文档目的

本文档不以论文写作为目标，而以**指导后续实验设计与实现落地**为目标。核心任务是把当前想法整理成一套可执行的研究路线，明确：

1. 这项工作的真正创新点是什么。
2. 哪些部分是必须先验证的核心假设。
3. 实验应当按什么顺序推进。
4. 哪些设计暂时不要做得过重，以避免工程失控。
5. 后续当实验结果成型后，如何自然升级为论文贡献。

---

## 2. 研究问题

### 2.1 背景

多模态大模型（MLLM）在处理高分辨率图像时，视觉 token 数量会迅速增加，导致：

- 计算量显著上升；
- 推理延迟增大；
- 小模型（如 1B 量级）难以承受高分辨率视觉输入；
- 简单降采样会丢失细节，从而影响复杂视觉问答、定位和 OCR 类任务。

现有高效方法通常采用：

- token pruning；
- token merging；
- 固定低分辨率输入；
- 基于启发式的 patch/region 选择。

这些方法的共同特点是：**围绕 token 的保留与丢弃来优化效率**。

### 2.2 核心矛盾

当前方法的本质限制不是“token 太多”本身，而是：

> 模型缺乏根据 query 和内容动态分配算力的能力。

换句话说，不是所有视觉 token 都应获得同样的计算深度；真正关键的是：

> 在给定计算预算下，哪些 token 应该获得更多计算，哪些 token 只需要浅层处理，哪些 token 可以跳过。

---

## 3. 核心创新点

## 3.1 一句话定义

本工作的核心创新不是 token pruning，而是：

> **在全局计算预算约束下，基于文本 query 对视觉 token 进行动态计算路径分配（compute routing）。**

---

## 3.2 与已有工作的根本区别

### 现有工作常见思路

- 保留重要 token，丢弃不重要 token；
- 合并相似 token；
- 对 token 做静态压缩；
- 根据 query 做 token 重要性打分。

### 本工作要强调的区别

本工作不把问题定义为“是否保留 token”，而把问题定义为：

> **每个 token 应该被分配到什么计算路径。**

例如，对一个 token，不是简单地做 keep/drop，而是让它进入如下路径之一：

- skip：跳过或仅保留占位表示；
- shallow：只经过轻量浅层编码；
- deep：经过更深层的精细编码。

因此，这个工作本质上是：

- 从 token selection 升级到 compute allocation；
- 从 heuristic filtering 升级到 budget-constrained optimization；
- 从静态统一计算升级到 query-adaptive dynamic computation。

---

## 3.3 当前阶段最稳妥的创新表述

为了兼顾新颖性与可实现性，当前阶段建议把创新点明确限定为：

### 创新点 1：Query-adaptive compute routing

根据文本 query 和粗粒度视觉 token，共同决定每个 token 的计算路径。

### 创新点 2：Budget-constrained optimization

不使用简单 top-k 或固定保留比例，而是通过显式计算预算约束，让模型学习在预算内分配计算资源。

### 创新点 3：Soft-to-hard routing

训练阶段采用可微分的软路由，推理阶段切换为硬路由，从而同时兼顾可训练性和真实推理效率。

### 创新点 4：Single-axis routing first

主实验先只做“深度维度”的路径分配，不同时引入高分辨率重编码和多维复杂控制，确保系统可实现、可解释、可复现。

---

## 4. 当前建议采用的技术路线

## 4.1 总体框架

推荐采用如下主干流程：

```text
Image → Coarse Encoder → Query-Adaptive Router → Multi-Path Depth Execution → Fusion → MLLM
```

其中：

- Coarse Encoder 负责用低成本提取全局视觉上下文；
- Router 负责根据 query 为 token 分配路径；
- Multi-Path Execution 负责执行不同深度的计算；
- Fusion 将不同路径输出的 token 汇总送入 MLLM。

---

## 4.2 当前版本只做单轴路径

为了避免工程实现过重，建议现阶段只保留三条路径：

```text
{skip, shallow, deep}
```

不要在第一阶段同时加入：

- high-resolution re-encoding；
- region crop；
- attention-level routing；
- 多轴联合控制。

原因很简单：

1. 第一阶段最重要的是验证核心假设是否成立；
2. 过多路径会极大增加实现复杂度；
3. 一旦实验不稳定，将很难判断问题来自哪一个模块；
4. reviewer 未来也更愿意接受“先验证一个清晰轴”的设计。

---

## 4.3 为什么先做深度路径最合适

相比 resolution routing，depth routing 更适合作为第一阶段：

- 不需要额外 crop / resize pipeline；
- 更容易直接在现有视觉 backbone 上改造；
- 更容易统计 FLOPs 和 latency；
- 更容易做 clean ablation；
- 更适合回答“compute allocation 是否有效”这个核心问题。

---

## 5. 方法设计原则

## 5.1 设计原则一：Router 必须极轻量

Router 不能成为新的计算瓶颈。

因此当前方案中，Router 的定位应该是：

- 一个小型 MLP；或
- 单层极简打分模块；
- 避免复杂多层 cross-attention。

### 实施约束

建议显式规定：

> Router 的额外计算量应尽量控制在总 FLOPs 的 5% 以内。

否则会出现“节省的算力不够覆盖路由自身开销”的反噬问题。

---

## 5.2 设计原则二：训练必须可微

如果直接对路径做 argmax 选择，会导致不可导、梯度断裂，训练不稳定。

因此建议采用：

- 训练阶段：soft routing 或 Gumbel-Softmax；
- 推理阶段：hard routing。

可以采用如下统一表达：

```text
训练时：f_i = Σ r_i^k · f_i^k
推理时：f_i = f_i^(argmax_k r_i^k)
```

这也是后续实现中最重要的技术细节之一。

---

## 5.3 设计原则三：渐进式扩展，而非一步到位

当前工作最容易失败的原因之一，不是 idea 不好，而是一次性把系统做得太复杂。

因此建议严格采用如下推进方式：

### 第一阶段

只做：

- coarse encoder；
- lightweight router；
- depth-only routing；
- task loss + compute regularization。

### 第二阶段

在第一阶段跑通并看到明确收益后，再尝试：

- 加入更丰富的路径；
- 尝试 resolution-aware routing；
- 做更复杂的 query interaction；
- 做多轴联合路由。

---

## 6. 需要验证的核心科学假设

后续所有实验都应围绕下列假设展开，而不是一开始追求最终 SOTA。

### 假设 H1

**不同视觉 token 对当前 query 的重要性不同，因此统一计算是低效的。**

### 假设 H2

**基于 query 的路径分配比仅基于图像内容的路径分配更有效。**

### 假设 H3

**在相同或相近 FLOPs 下，compute routing 能比 pruning 获得更优性能。**

### 假设 H4

**soft-to-hard routing 能在保证训练稳定的同时带来实际推理收益。**

### 假设 H5

**即使只在深度维度上做路径分配，也足以带来可测量的效率—性能收益。**

---

## 7. 实验推进顺序

后续实验必须按层次推进，而不是一开始就追求完整系统。

## 7.1 Phase 0：最小可运行原型

目标：先证明代码能跑通。

### 建议配置

- 选择一个现成 MLLM backbone；
- 保持视觉主干基本不改；
- 在视觉 token 进入主干前或中间层加入 depth router；
- 路径只做三类：skip / shallow / deep。

### 验证点

- forward 是否稳定；
- soft routing 是否能正常反传；
- router 输出是否不是塌缩的；
- FLOPs 统计是否清晰。

### 结果要求

这一阶段不追求性能最优，只要求：

- 训练稳定；
- 计算路径可视化合理；
- 能测出不同预算下的行为变化。

---

## 7.2 Phase 1：核心假设验证

目标：回答“这个创新点到底值不值得继续做”。

### 必做对比

1. 无 routing 的原始 baseline；
2. image-only routing；
3. query-adaptive routing；
4. hard routing vs soft routing；
5. 不同 budget。

### 核心问题

- query conditioning 是否真的有用；
- compute regularization 是否真的控制了算力；
- routing 是否只是学会全部走 deep。

---

## 7.3 Phase 2：与 pruning 的公平对比

目标：回答“这是否只是另一种 pruning”。

### 核心设计

必须尽量在相近 FLOPs 或 latency 下对比：

- full token baseline；
- low-resolution baseline；
- pruning baseline；
- 本方法。

### 核心观察

如果在相同计算预算下，本方法性能更高，则可以支持你的核心论点：

> 真正重要的不是减少 token 数，而是更合理地分配计算。

---

## 7.4 Phase 3：增强版与扩展版

只有当前两阶段效果明确后，才建议做：

- 更复杂 router；
- 更多路径；
- 局部高分辨率重编码；
- 不同任务泛化实验；
- 更大模型或更大数据集上的验证。

---

## 8. 建议优先做的实验清单

## 8.1 最优先实验

### E1. Routing 是否能节省计算

观察：

- 不同 budget 下的 FLOPs；
- 不同 budget 下的 latency；
- deep path 使用比例。

### E2. Query conditioning 是否必要

对比：

- image-only router；
- query-adaptive router。

### E3. Compute routing 是否优于简单 pruning

对比：

- 相同 FLOPs 下的性能；
- 相同 latency 下的性能。

### E4. 训练—推理协议是否成立

对比：

- train-soft / infer-hard；
- train-hard 近似方案；
- 观察训练稳定性和收敛速度。

---

## 8.2 强烈建议做的可视化

### 可视化 1：不同 query 下的路径分配热图

用于说明：

- 同一张图，不同 query 会导致不同 token 获得不同计算深度。

### 可视化 2：Budget 变化下的路径分配比例

用于说明：

- 模型是否真的学会在预算约束下压缩计算。

### 可视化 3：错误样例分析

用于判断：

- 是否因为 router 错判，导致关键 token 被过早跳过。

---

## 9. 当前阶段不建议过早做的内容

为了提高成功率，以下内容建议暂时不要作为第一优先级：

### 9.1 不要先做 multi-axis routing

即不要一开始同时做：

- 深度路由；
- 高分辨率重编码；
- attention sparsification。

### 9.2 不要先追求“顶会式全能系统”

当前最重要的不是堆功能，而是证明核心论点成立。

### 9.3 不要先写成 pruning narrative

如果一开始就把它讲成“动态选择重要 token”，后续无论实验还是论文都很容易被归到已有 pruning 工作里。

应始终坚持如下叙述：

> 本工作研究的是 compute allocation，而不是 token elimination。

---

## 10. 风险清单与对应策略

## 风险 1：Router 开销过大

### 表现

- 虽然深路径 token 变少了，但总 FLOPs 没降多少；
- latency 没改善，甚至更差。

### 解决策略

- 把 router 控制为 tiny MLP；
- 避免复杂 cross-attention；
- 单独报告 router overhead。

---

## 风险 2：Routing collapse

### 表现

- 所有 token 都走同一路径；
- deep path 占比过高或过低。

### 解决策略

- entropy regularization；
- load balancing；
- temperature annealing；
- 合理初始化 budget penalty。

---

## 风险 3：训练不稳定

### 表现

- routing 波动大；
- loss 不收敛；
- Gumbel 噪声过大。

### 解决策略

- 先用 soft routing 验证；
- Gumbel-Softmax 作为后续增强；
- 初期增大温度，后期再退火。

---

## 风险 4：创新点被误解为 pruning

### 表现

- 实验中只汇报 token 数减少；
- 方法描述以 selection 为中心。

### 解决策略

- 始终报告 compute budget、FLOPs、latency；
- 做“同 FLOPs 下优于 pruning”的对比；
- 明确路径分配而非简单删 token。

---

## 风险 5：系统过重导致开发卡死

### 表现

- 分支太多；
- 代码耦合严重；
- 很久都跑不通第一版。

### 解决策略

- 保持 single-axis routing；
- 先做 depth-only 原型；
- 把所有复杂扩展放到第二阶段。

---

## 11. 当前最推荐的项目定义

如果你现在就要开始做实验，我建议把项目定义成下面这一句：

> **在固定计算预算下，研究基于 query 的视觉 token 深度路径分配是否能够优于传统 token pruning，并带来更好的效率—性能折中。**

这个定义有几个优点：

- 足够聚焦；
- 足够新；
- 足够容易实现第一版；
- 后续容易扩展成更大的框架。

---

## 12. 后续执行建议

当前最合理的执行顺序如下：

### 第一步

确定 backbone 与插入 routing 的位置。

### 第二步

先实现 depth-only 的最小原型。

### 第三步

完成最基础的 4 组实验：

- baseline；
- image-only routing；
- query-adaptive routing；
- 不同 budget。

### 第四步

确认是否存在稳定的效率—性能收益。

### 第五步

如果结果成立，再决定是否扩展到：

- 更复杂 router；
- 更强 baseline；
- 多任务；
- 多维 routing。

---

## 13. 最终结论

当前这项工作的真正价值，不在于发明一种新的 token 筛选技巧，而在于提出并验证下面这个核心命题：

> **对于多模态模型，高效视觉理解的关键不只是减少 token，而是在 query 条件下对 token 分配不同程度的计算。**

因此，后续所有实验都应服务于这一命题的验证。

如果实验支持这一点，那么后续无论写论文、做系统扩展，还是加入高分辨率路径，都会变得顺理成章。
