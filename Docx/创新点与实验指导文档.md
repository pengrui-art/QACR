# Query-Adaptive Compute Routing 研究创新点与实验指导文档

## 1. 这份文档现在的用途

这份文档不再服务于“从零定义项目”，而是服务于当前阶段两个目标：

1. 明确 QACR 现在还成立的核心创新点。
2. 指导后续实验按最稳、最有效的顺序推进。

---

## 2. 当前项目的最准确问题定义

QACR 现在最准确的研究问题是：

> **在固定计算预算下，基于 query 为视觉 token 分配不同计算深度，是否能够比统一计算或简单 pruning 带来更好的效率-性能折中。**

这里的关键词必须固定：

- `query-adaptive`
- `compute routing`
- `budget-constrained`
- `depth-only first`

不要再把主问题写成“动态选重要 token”，否则很容易滑回 pruning 叙事。

---

## 3. 当前仍然成立的核心创新点

### 创新点 1：Query-adaptive compute routing

路由决策同时依赖：

- 文本 query
- 粗粒度视觉 token

所以模型不是固定地偏好文字区或物体区，而是根据问题动态决定哪里更值得深算。

### 创新点 2：从 token selection 转向 compute allocation

QACR 的核心不是 keep / drop，而是给 token 分配不同路径：

- `skip`
- `shallow`
- `deep`

因此它更适合被定义为：

> compute allocation，而不是 token elimination

### 创新点 3：Budget-constrained execution

QACR 不是只追求效果，也不是只追求省算，而是在目标预算附近学习如何分配计算。

### 创新点 4：Soft-to-hard routing

训练阶段用软路由保证可学；
推理阶段切换到硬路由，并尽量贴近预算执行。

---

## 4. 当前不该再当主创新推进的方向

基于现有实验，以下方向不应继续作为主创新去讲：

- 高分辨率重编码
- 多轴联合 routing
- `min_keep_ratio / min_deep_ratio` 硬阈值保留支线

原因很明确：

1. 目前没有稳定且可复现的增益。
2. 会让主线从“清晰的单轴方法”变成“功能堆叠”。
3. 不利于论文叙事，也不利于实验判断。

这些方向可以保留为 future work，但不应继续占主线资源。

---

## 5. prompt / 后处理 / 评测修复的定位

这部分非常重要，但不应被当成方法本体的第一创新点。

更准确的定位是：

> OCR / 文档任务上的实验工程、评测协议和 inference calibration

当前已经验证出的稳定结论：

- `TextVQA` 和 `DocVQA` 需要分数据集 prompt 策略
- 后处理必须保护：
  - 小数
  - 时间
  - 日期
  - 金额
  - URL / email
  - 人名缩写与首字母
- 评分必须使用正确的数据集口径

当前建议：

- `TextVQA`：使用更强的“只输出答案、不要复述问题”约束
- `DocVQA`：保持保守 exact-copy prompt

---

## 6. 最新实验结论

### 6.1 小样本结论

已经验证：

- 更强的硬保留参数 sweep 基本无效
- OCR / 文档任务上的 prompt 与答案抽取调整是有效方向
- `TextVQA` 和 `DocVQA` 不适合强行共用一套 prompt

### 6.2 最新串行全量结果

相对上一版主结果：

- `TextVQA`: `0.27133 -> 0.25793`
- `DocVQA`: `0.10077 -> 0.11984`

对应解读：

- `DocVQA` 明显改善，说明保守 exact-copy + 新抽取规则是对的
- `TextVQA` 下滑，说明 scene-text 题上的 prompt / 抽取仍然没有调稳

### 6.3 当前最重要的判断

当前主线不是“方法完全失效”，而是：

> **QACR 的方法主线成立，但 OCR / 文档专项执行策略仍然强烈影响最终主表结果。**

---

## 7. 当前阶段实验优先级

### 第一优先级：保证主表可信

先保证：

- 评测口径正确
- 结果稳定落盘
- 全量 benchmark 可复现

### 第二优先级：单变量小样本验证

任何新改动都先用小样本验证，再决定是否推全量。

### 第三优先级：只推进已显示正信号的方向

当前值得继续推进的方向：

- `TextVQA` 的 prompt / 抽取修正
- `DocVQA` 的 exact-copy 口径保持
- 更 query-aware 的 routing signal

当前不值得继续大规模扫参的方向：

- 更强 `min_keep_ratio / min_deep_ratio`
- 高分辨率重编码

---

## 8. 当前推荐实验流程

### Step 1：锁定协议

先固定：

- 数据集
- 评分口径
- prompt 版本
- postprocess 版本

### Step 2：小样本试验

推荐先跑：

- `max-samples=200`
- 单变量改动
- 同时记录 accuracy 与 compute

### Step 3：确认方向后再推全量

只有当小样本出现清晰正收益时，才推进全量 benchmark。

### Step 4：全量 benchmark 串行执行

当前机器上，8B 全量评测不建议并发。

稳定原则写死为：

- 小样本可以多卡并行
- 全量 benchmark 尽量串行

当前推荐脚本：

- `scripts/run_phase64_promptsplit_serial.sh`

## 10. 当前建议补的实验

接下来最值得补的不是“更复杂系统”，而是下面两类：

### A. TextVQA 专项提效

重点看：

- 名称识别
- 品牌识别
- 数字读取
- scene-text 精确抄写

### B. Routing 信号提效

重点看：

- 如何让 router 更稳定地区分“真正需要 deep 的 token”
- 而不是靠硬阈值强保
