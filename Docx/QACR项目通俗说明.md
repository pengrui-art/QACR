# QACR 项目通俗说明

## 1. 一句话讲明白

QACR 想做的事情是：

> 让多模态模型看图回答问题时，不是对整张图平均用力，而是根据问题，把算力重点花在真正相关的区域上。

同一张图，问题不同，重点计算的位置也应该不同。

---

## 2. 我们到底在解决什么问题

大模型看图时有一个现实矛盾：

- 图越清晰，视觉 token 越多；
- token 越多，计算越贵；
- 如果粗暴缩图，又会丢掉细节，尤其伤 OCR、文档理解和细粒度问答。

所以关键问题不是“要不要看图”，而是：

> 有限算力到底该花在哪些视觉 token 上。

这就是 QACR 的出发点。

---

## 3. 和传统方法的区别

很多高效方法主要在回答：

> 哪些 token 要保留，哪些 token 要丢掉？

而 QACR 关注的是另一个问题：

> 每个 token 应该获得多少计算？

所以它的核心不是简单 pruning，而是：

> **compute allocation（算力分配）**

具体来说，我们让 token 走三条不同路径：

- `skip`：几乎跳过
- `shallow`：浅层处理
- `deep`：重点深算

这比单纯 keep / drop 更细，也更符合“按题目分配算力”的直觉。

---

## 4. 现在项目真正成立的创新点

截至目前，项目的**核心创新没有变**，但主线已经明显收束。

### 创新点 1：Query-Adaptive Compute Routing

路由不是只看图像内容，还会结合问题内容。

通俗地说就是：

> 问什么，就重点算什么。

### 创新点 2：三路径算力分配

我们不是只做 token 删除，而是让每个 token 获得不同深度的计算：

- `skip`
- `shallow`
- `deep`

### 创新点 3：预算约束

QACR 不是“想怎么算就怎么算”，而是在给定预算附近分配算力。

比如预算 `0.45`，就希望整体大致只使用完整深算力的 `45%` 左右。

### 创新点 4：Soft-to-Hard 路由

训练时用软路由，保证可学习；
推理时切成硬路由，保证真实执行时路径明确。

---

## 5. 哪些东西现在不该再当主创新讲

我们已经做过不少扩展探索，但从当前结果看，不适合继续当“主创新”去讲的有：

- 高分辨率重编码
- 多轴联合 routing
- 通过 `min_keep_ratio / min_deep_ratio` 硬保关键 token

原因不是它们完全没价值，而是：

- 目前没有稳定收益；
- 会把主线讲散；
- 不利于论文叙事和实验判断。

所以现在最稳的项目定义是：

> **QACR 的核心创新是：在预算约束下，基于 query 为视觉 token 分配不同计算深度。**

---

## 6. OCR / 文档里的 prompt 和后处理算不算创新

这部分很重要，但更准确的定位不是“方法本体创新”，而是：

> 让 QACR 在 OCR / 文档任务上被公平评测、正确发挥出来的实验工程与评测协议。

我们已经验证过：

- `TextVQA` 和 `DocVQA` 不能生硬共用同一套 prompt；
- 后处理必须保护小数、时间、日期、金额、人名缩写等格式；
- 评测口径会真实影响最终分数。

当前较稳定的经验是：

- `TextVQA` 适合更强的“只输出答案、不要复述问题”约束；
- `DocVQA` 更适合保守的 exact-copy 风格提示。

---

## 7. 这次最新全量结果说明了什么

在最近一轮串行全量复跑中，我们拿到了新的有效结果：

- `TextVQA`: `0.27133 -> 0.25793`
- `DocVQA`: `0.10077 -> 0.11984`

这说明当前策略的效果是分化的：

- `DocVQA` 明显回升，说明保守 exact-copy 提示 + 新的 question-aware 抽取是有效的；
- `TextVQA` 反而下降，说明我们现在这版 prompt / 抽取对 scene-text 题仍然过于激进。

所以现在最准确的判断是：

> **QACR 在 OCR / 文档任务上的实验工程仍然是关键影响项，而且不同数据集需要分开调。**

---

## 8. 当前项目处在什么阶段

可以很诚实地说，项目已经走过“想法验证”阶段，但还没有到“结果足够强可以直接投稿”的阶段。

已经成立的部分：

- 系统能跑通；
- 路由很轻量；
- query-aware routing 这条主线成立；
- 和 pruning 不同的叙事是清楚的；
- 全量 benchmark 已经能稳定跑通。

还没完全解决的部分：

- OCR / 文档效果还不够强；
- 关键 token 保护还不够准；
- 真实 latency 优势还不够扎实；
- 主表结果距离强基线仍有差距。

---

## 9. 现在最值得继续做什么

当前最值得继续做的，不是继续扩系统，而是继续把主线效果拉起来：

1. 把 `TextVQA` 的 prompt / 抽取继续调稳。
2. 保持 `DocVQA` 的 exact-copy 策略。
3. 继续做更 query-aware 的 routing 信号，而不是继续扫硬阈值。
4. 在稳定主表后，再考虑 latency 和更复杂扩展。

---

## 10. 最后一句话总结

QACR 现在最准确的定义是：

> **它不是“又一个删 token 的方法”，而是一个在预算约束下、根据问题动态分配视觉算力的框架。**

这个核心创新仍然成立；现在真正要继续攻的，是把这条主线在正式 benchmark 上做得更强、更稳。
