# QACR 项目通俗说明

## 1. 一句话先讲明白

我们的项目想做的是：

> **让多模态大模型在看图回答问题时，不要对整张图一视同仁地“平均用力”，而是根据问题内容，把算力重点花在真正相关的区域上。**

比如同一张图：

- 如果问题是“左边那个物体是什么？”
- 模型就应该把更多计算放在左边。

如果问题变成：

- “底部写了什么文字？”
- 模型就应该把更多计算放在底部文字区域。

这就是我们项目最核心的想法。

---

## 2. 我们到底在解决什么问题

现在的大模型看图时，经常会遇到一个很现实的问题：

- 图片一大，视觉 token 就很多；
- token 一多，计算就很贵；
- 计算一贵，推理就会变慢，显存也会变大；
- 如果粗暴把图缩小，又会丢掉细节，尤其影响 OCR、文档理解、细粒度视觉问答。

所以问题不是“图像不能看”，而是：

> **模型到底应该把有限的计算资源花在哪里。**

这就像考试时间有限，不可能每道题都花一样多的时间。真正聪明的做法应该是：

- 重点题多花时间；
- 简单题少花时间；
- 完全无关的内容可以快速略过。

我们就是把这个想法搬到了多模态模型里。

---

## 3. 传统方法一般怎么做

很多已有方法在做高效视觉计算时，常见思路是：

- 直接删掉一部分 token；
- 合并一些相似 token；
- 把整张图统一缩小；
- 用一些启发式规则，只保留“看起来重要”的部分。

这些方法并不是没用，但它们大多在回答一个问题：

> “哪些 token 要保留，哪些 token 要丢掉？”

而我们的项目想回答的是另一个更细的问题：

> “每个 token 应该获得多少计算？”

这两者看起来很像，其实区别很大。

传统方法更像：

- 留下谁；
- 删掉谁。

我们的方法更像：

- 这个区域值不值得深度处理？
- 这个区域只需要浅看一下吗？
- 这个区域是不是几乎不用算？

也就是说，我们不是简单做“保留/删除”，而是在做：

> **算力分配（compute allocation）**

这也是我们项目最重要的创新视角。

---

## 4. 我们的创新点是什么

如果用最通俗的话来讲，我们的创新主要有 4 点。

### 创新点 1：模型会“看问题分配算力”

我们不是只看图片内容来决定哪里重要，而是同时看：

- 图像内容；
- 用户提出的问题。

也就是说，模型不是固定地认为“文字区域永远最重要”或者“物体边缘永远最重要”，而是会根据当前 query 去决定。

这就是：

> **Query-Adaptive Compute Routing**

通俗理解就是：

> 问什么，就重点算什么。

### 创新点 2：我们不是只做“删 token”，而是给 token 分配不同计算路径

我们为每个视觉 token 设计了三条计算路径：

- `skip`：几乎跳过，不花太多算力；
- `shallow`：简单处理一下；
- `deep`：重点深度处理。

所以我们不是只判断“留不留”，而是判断：

> **这个 token 值得被算到什么程度。**

这比传统 pruning 更细，也更灵活。

### 创新点 3：我们给模型一个“预算”

模型虽然可以动态分配算力，但不能无限制地用算力。

所以我们加了一个很重要的约束：

> **总计算量必须尽量控制在目标预算附近。**

比如预算设成 `0.45`，意思就是希望模型大概只用完整深算力的 45% 左右。

这样做的好处是：

- 方法不会失控；
- 不会一边说省算力，一边偷偷全走深路径；
- 可以系统地研究“算力和效果之间怎么折中”。

### 创新点 4：训练时柔和，推理时果断

如果一上来就让模型硬性选择：

- 这个 token 走 skip；
- 那个 token 走 deep；

训练会很难，因为这种离散选择不好反向传播。

所以我们采用了一个更稳妥的策略：

- 训练时先用“软路由”，让模型可以平滑学习；
- 推理时再切换成“硬路由”，真正做出离散决策。

这就是：

> **Soft-to-Hard Routing**

通俗理解就是：

> 练习时可以模糊一点，考试时必须明确做选择。

## 5. 我们具体是怎么做的

整个系统可以用下面这条链路理解：

```text
图片 + 问题
    -> 提取粗粒度视觉 token
    -> 路由器判断每个 token 该走哪条路
    -> 不同 token 走不同深度的计算路径
    -> 融合结果
    -> 输出给后续模型做回答
```

下面把它拆开讲。

### 第一步：先把图像变成一批粗粒度 token

模型先不会立刻对整张图做特别昂贵的深处理，而是先提取一层较粗的视觉表示。

你可以把它理解成：

- 先大致扫一眼整张图；
- 知道每个区域大概是什么；
- 为后面的“算力分配”做准备。

### 第二步：让路由器决定每个 token 走哪条路径

然后我们有一个很轻量的小模块，叫 Router。

它会同时看：

- 当前问题在问什么；
- 每个视觉 token 大概是什么内容。

然后输出这个 token 更适合：

- `skip`
- `shallow`
- `deep`

这一步就像一个调度员，在决定：

- 哪些地方需要重点看；
- 哪些地方只要扫一眼；
- 哪些地方先别花太多时间。

### 第三步：真的按不同路径去算

路由器做完决定后，不同 token 会进入不同的计算路径：

- `skip` 路径：几乎不做额外处理；
- `shallow` 路径：只做轻量计算；
- `deep` 路径：做更充分、更精细的处理。

这样一来，模型就不是“全图统一深算”，而是“重要区域精算，不重要区域省算”。

### 第四步：用预算约束住整体算力

为了防止模型偷懒或者乱花算力，我们会在训练中加入一个预算损失。

它的作用是：

- 如果模型算得太多，就提醒它收一点；
- 如果模型过度偷工减料，也会被任务损失拉回来。

所以模型最终学到的是一种平衡：

> 在尽量省算力的前提下，把效果做得尽可能好。

### 第五步：训练和推理分开处理

训练时我们用软路由，让模型能学。

推理时我们用硬路由，让模型真正按照离散决策执行。

这使得整套方法既能训练起来，又有机会在推理时得到真正的动态计算收益。

## 6. 举个最直观的例子

假设有一张图，左边有一个蓝色物体，底部有一行文字。

如果问题是：

> “左边那个物体是什么？”

那模型更合理的做法应该是：

- 左边区域走 `deep`
- 其他区域走 `shallow` 或 `skip`

如果问题换成：

> “底部写了什么字？”

那模型应该把更多计算分给底部文字区域，而不是继续重点看左边物体。

这就是我们项目的核心价值：

> **同一张图，因为问题不同，计算重点也应该不同。**

这比“整张图统一深算”更聪明，也比“固定删一部分 token”更灵活。

## 7. 目前我们已经做到什么程度了

从目前实验来看，这个方向已经不是纸上谈兵，而是真的做出了一些有说服力的结果。

### 7.1 已经证明的部分

#### 1. 系统能跑通

我们已经完成了：

- 基座模型跑通；
- Router 跑通；
- 三条路径跑通；
- soft-to-hard 训练链路跑通；
- 预算约束跑通；
- 端到端稳定性测试跑通。

也就是说，这不是一个停留在概念图上的方案，而是一个已经完成原型实现的系统。

#### 2. Router 很轻量

我们测到 Router 的额外开销只有：

`0.008070%`

这意味着 Router 本身几乎不会成为新瓶颈。

#### 3. 模型真的学会了“看问题分配计算”

不同 query 下，热图明显不同，说明模型不是固定地关注同一块区域，而是真的会根据问题变化计算重心。

#### 4. 相比传统 pruning，我们的方法更有希望

在近预算对比里，QACR 的 proxy task loss 明显优于传统 token pruning。

这说明：

> **“动态分配算力”这条路，至少比“简单删 token”更值得继续做。**

### 7.2 还没有完全解决的部分

我们也必须诚实地说，目前还有几块短板。

#### 1. 关键 token 还保护得不够好

在 corner case 分析里，我们发现：

- 一些真正关键的区域，仍然没有拿到足够深的计算；
- 也就是说，模型虽然会“动态变化”，但还没有做到“动态得很准”。

#### 2. 真实硬件加速还没有完全证明出来

从目前 profiling 看，方法在机制上有潜力，但真实端到端 latency 优势还没完全打出来。

原因主要是：

- 当前执行器还不是完全理想的稀疏实现；
- 动态路由带来的索引和回填开销还存在；
- 一些底层加速算子还没有充分利用。

#### 3. 目前还主要是 proxy 实验

我们已经做了比较完整的原型验证，但要成为真正成熟的论文，还需要更多官方 benchmark 上的大规模结果。

---

## 8. 为什么这个项目值得继续做

虽然现在还不是完全体，但这个项目已经显示出几个很有价值的特点。

### 第一，它的研究问题是对的

现在多模态模型越来越大，视觉输入越来越贵。

“怎么更聪明地分配算力”本身就是一个非常重要的问题。

### 第二，它的创新点是清楚的

我们不是在重复做传统 pruning，而是在提出一种新的思路：

> **从 token selection 走向 compute allocation**

这点在论文叙事上是很重要的。

### 第三，它已经有初步数值支撑

现在不是“感觉上好像有道理”，而是已经出现了：

- 路由热图变化；
- 预算响应；
- 与 pruning 的近预算对比优势；
- 稳定训练结果。

这些都说明它值得继续深入。

---

## 9. 你可以把这个项目理解成什么

如果要用最生活化的比喻，我会这样形容：

> 以前的模型像一个不管题目问什么、都把整张图从头到尾认真看一遍的人。

而我们的模型更像一个会读题的人：

- 先看问题在问什么；
- 再决定重点看哪里；
- 对重要区域仔细看；
- 对不重要区域快速扫过；
- 在总时间有限的前提下，把注意力和算力花在最值钱的地方。

这就是 QACR 的本质。

---

## 10. 最后用最简单的话总结

我们这个项目的创新，不是“删掉一些 token”这么简单，而是：

> **让模型根据问题，动态决定图像中不同区域应该获得多少计算。**

具体做法是：

1. 先把图像转成粗粒度 token；
2. 用一个轻量 Router 结合问题和图像内容，判断每个 token 该走哪条计算路径；
3. 让不同 token 分别走 `skip / shallow / deep`；
4. 用预算约束住整体算力；
5. 训练时用软路由，推理时用硬路由。

目前我们已经证明：

- 这件事能做；
- 方向是对的；
- 它比传统 pruning 更有潜力；
- 但它还没有完全打磨到最终形态。

所以最准确的评价是：

> **这是一个已经被验证“值得继续做”的创新方向。**
